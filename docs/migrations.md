# Database migrations (Alembic)

The schema is managed by **Alembic** (async), not `metadata.create_all`. Tests and
the e2e seeder build their schema by running the real migrations (via
`app/db/migrate.py`), so the migrations stay honest and dev/test schema matches
production.

## Changing the schema

Edit the ORM models in [app/db/models.py](../app/db/models.py), then generate and
apply a migration:

```bash
just migration "describe the change"   # writes migrations/versions/<rev>.py
just migrate                           # apply to the configured DB
```

These wrap `uv run alembic revision --autogenerate -m "…"` and `uv run alembic
upgrade head` respectively. Review the autogenerated revision before committing it
(autogenerate isn't perfect), and **commit it alongside the model change**. The
gate (`just check`) runs `alembic check`, which fails if `models.py` has drifted
ahead of the migrations — so a model edit without its migration won't pass.

## Configuration

The connection URL comes from app settings (`DATABASE_URL`) inside
`migrations/env.py` — never hard-code credentials in `alembic.ini`. Alembic's
`env.py` and generated revisions are exempt from `ruff format` (see
`pyproject.toml`) so autogenerate doesn't fight the formatter.
